# TripSync PRD (Product Requirements Document)

> **서비스명:** TripSync (트립싱크)
> **문서 버전:** v1.1
> **작성일:** 2026-04-02
> **최종 수정:** 2026-04-03
> **상태:** MVP 개발 단계

---

## 1. 문서 개요

### 1.1 목적

본 문서는 TripSync의 MVP(Minimum Viable Product) 개발에 필요한 제품 요구사항을 정의한다. 한국관광공사 관광 데이터 공모전 출품을 목표로 하며, 개발팀이 구현 범위와 우선순위를 명확히 이해할 수 있도록 작성한다.

### 1.2 핵심 한줄 정의

> **"갈등 없는 여행이 지방을 살린다"** — 동행자 간 취향 충돌이 해결되면, 그룹은 비로소 유명 관광지 너머 지방의 숨은 명소까지 닿을 수 있다. TripSync는 그룹 합의를 만들고, 그 결과로 인구 감소지역에 사람이 흐른다.

### 1.3 용어 정의

| 용어 | 정의 |
|------|------|
| **TPTI** | Trip Personal Type Indicator. 4개 축(활동성, 사진, 예산, 테마)으로 여행 취향을 정량화하는 도구 |
| **합의 엔진** | 동행자 간 TPTI 벡터를 분석하여 시간 배분 기반의 최적 일정을 생성하는 핵심 알고리즘 |
| **갈등 지도** | 동행자들의 TPTI 결과를 겹쳐 충돌 축과 공통 지대를 시각화한 리포트 |
| **공통 지대** | 모든 동행자가 유사한 점수를 보이는 TPTI 축. 일정의 기본 배경으로 확정됨 |
| **충돌 축** | 동행자 간 점수 편차가 큰 TPTI 축. 시간 배분(Time Slicing)으로 교대 반영함 |
| **방장** | 여행 그룹을 생성하고 동행자를 초대하는 사용자 |
| **슬롯** | 일정 내 시간 단위 블록 (최소 1시간, 최대 3시간) |
| **인구감소지역** | 행정안전부 지정 89개 시·군. 일정 생성 시 해당 지역 장소에 가중치를 부여하여 자연스러운 방문을 유도 |
| **앵커 장소** | 유명 관광지. 인구감소지역 숨은 명소와 조합하여 일정의 구심점 역할 |
| **숨은 명소** | 인구감소지역 내 TourAPI 데이터 기반 비교적 덜 알려진 장소. 앵커 장소와 동선상 묶여 추천됨 |

## 2. 문제 정의

### 2.1 현재 상황

여행 장소 정보와 추천 서비스는 이미 시장에 충분히 존재한다 (트리플, 마이리얼트립, 네이버 여행 등). 그러나 이 서비스들은 모두 **개인 맞춤 추천**에 초점이 맞춰져 있으며, 결과적으로 유명 관광지에만 편중된 여행을 반복시킨다.

그 이면에는 해결되지 않은 근본 문제가 있다.

### 2.2 핵심 문제: 갈등이 여행을 좁힌다

```
그룹 여행 = 의견 충돌이 두렵다
       ↓
"검증된 유명 관광지만 가자" — 갈등을 피하는 가장 쉬운 방법
       ↓
지방 소외 심화 — 알려진 곳만 반복, 인구감소지역 숨은 명소는 닿지 못함
```

**그룹 취향 충돌이 해결되지 않기 때문에, 여행자는 안전한 선택만 반복한다.**

구체적인 Pain Point:
- 한 명이 독박으로 일정을 짜고, 나중에 "난 별로인데?" 소리를 듣는 상황
- "가서 정하자"로 넘어갔다가 여행 첫날부터 갈등이 시작되는 상황
- 결국 모두가 아는 유명 장소로만 수렴 — 지방의 다양성은 사라짐

### 2.3 기존 솔루션의 한계

| 기존 서비스 | 한계 |
|------------|------|
| 트리플, 마이리얼트립 | 개인 맞춤 추천만 제공. 그룹 조율 기능 없음. 유명 장소 편중 |
| 카카오톡 투표 | "A 장소 vs B 장소" 단순 투표만 가능. 복합 취향 조율 불가 |
| 구글 스프레드시트 | 수동 조율. 방장이 독박으로 정리하게 됨 |
| 지자체 관광 앱 | 단순 장소 나열. 그룹 취향 반영 불가. 인구감소지역 연계 없음 |

## 3. 타겟 사용자

### 3.1 Primary Target

**"총대 메기 싫은 여행 모임의 방장"**

- 20~30대 그룹 여행 기획자
- 보통 친구/동료 3~5명과 여행을 가는 사람
- 일정 짜는 책임을 지고 싶지 않지만, 아무도 안 하니까 맡게 되는 사람

### 3.2 Secondary Target

- 그룹 여행 동행자 (방장이 보낸 링크로 참여)
- 커플 여행 계획자 (2인 그룹)

### 3.3 서비스 포지셔닝

TripSync는 **그룹 합의 문제를 해결하는 도구**다. 합의가 만들어지면, 그 결과로 지방이 살아난다.

| 관점 | 포지셔닝 |
|------|---------|
| **사용자** | "내가 짠 게 아니라 우리 TPTI 결과로 AI가 합의해서 짠 거야" — 방장의 민주적 책임 회피 도구 |
| **지역사회** | 합의된 그룹이 비로소 유명 관광지 너머를 탐색 → 인구감소지역 숨은 명소에 자연스럽게 사람이 흐름 |
| **공모전** | 그룹 갈등 해결(기획력·독창성) + 인구감소지역 연계(지역 특화 가점) + TourAPI 심층 활용(데이터 활용) |

**핵심:** 인구감소지역 방문은 "캠페인"이 아니다. 취향에 맞아서 가는 것이다. TripSync는 그 조건을 만든다.

## 4. 기능 명세 (MVP)

MVP는 **3화면 완결 구조**로, 각 화면이 독립적 가치를 제공하면서 순차적으로 연결된다.

### 4.1 화면 1: TPTI 검사

**목적:** 동행자의 여행 취향을 4개 축으로 정량화한다.

**TPTI 4축 정의:**

| 축 | 코드 | 양극단 | 설명 |
|----|------|--------|------|
| 활동성 (Mobility) | M | Walker(W) ↔ Stay(S) | 뚜벅이 체험형 vs 휴식 호캉스형 |
| 기록 (Photo) | P | Artist(A) ↔ Eyes(E) | 인생샷 SNS형 vs 눈으로 담는 실속형 |
| 예산 (Budget) | B | Luxury(L) ↔ Cost-effective(C) | 가심비 플렉스형 vs 가성비 절약형 |
| 테마 (Theme) | T | Nature(N) ↔ City(C) | 자연 힐링형 vs 도심 핫플형 |

**포지셔닝 원칙:** TPTI는 심리 검사가 아니라 **"여행 취향 대화의 구조화 도구"**이다. 정확한 분류보다 동행자 간 상대적 차이를 비교 가능하게 만드는 것이 목적이다.

**기능 상세:**

| ID | 기능 | 설명 | 우선순위 |
|----|------|------|:---:|
| F1-1 | 문항 응답 | 8개 문항 (축당 2문항), 30초 이내 완료 목표 | P0 |
| F1-2 | 점수 산출 | 각 축 0~100 연속 점수로 산출 | P0 |
| F1-3 | 결과 카드 | 레이더 차트(4축) + 캐릭터 별명 자동 생성 | P0 |
| F1-4 | SNS 공유 | 결과 카드 이미지를 인스타/카톡으로 공유 | P1 |
| F1-5 | 수치 수동 조정 | 검사 후 사용자가 직접 점수를 미세 조정 | P1 |

**TPTI 8문항 (확정):**

| 번호 | 문항 | 축 | 고점(100) 선택지 | 저점(0) 선택지 |
|:---:|------|:---:|------|------|
| Q1 | 여행 첫날, 숙소 도착 후 당신은? | M(활동성) | 바로 근처 마을 탐방 나선다 | 숙소에서 쉬며 내일을 아낀다 |
| Q2 | 여행 중 예상치 못한 골목을 발견했다면? | M(활동성) | 무조건 들어가 본다 | 일정대로 가는 게 편하다 |
| Q3 | 멋진 풍경을 마주쳤을 때 당신은? | P(기록) | 인생샷 찍으려 자리 잡는다 | 그냥 눈으로 담는다 |
| Q4 | 여행 끝나고 SNS에 올릴 사진은? | P(기록) | 최대한 많이 찍어둔다 | 한두 장만 찍으면 충분하다 |
| Q5 | 맛집 vs 가성비 식당, 선택은? | B(예산) | 비싸도 유명 맛집을 간다 | 저렴하고 든든한 곳이 좋다 |
| Q6 | 여행 기념품을 고를 때 당신은? | B(예산) | 좋으면 망설임 없이 산다 | 실용적인 것만, 최대한 아낀다 |
| Q7 | 여행지에서 더 끌리는 테마는? | T(테마) | 역사·문화 탐방 (유적지, 박물관) | 자연·힐링 (해변, 산, 공원) |
| Q8 | 하루 일정의 분위기는? | T(테마) | 카페·맛집·핫플 위주 도심형 | 걷기·경치·자연 위주 힐링형 |

> 점수 산출: 선택지마다 0~100 점수(5단계 리커트)를 부여, 축당 2문항 평균이 최종 축 점수

**캐릭터 별명 매핑 규칙 (4축 이진 분류, 기준점 50):**

| M | P | B | T | 별명 | 설명 |
|:---:|:---:|:---:|:---:|------|------|
| W | A | L | C | 🏙️ 감성 플렉서 | 도심 핫플에서 인생샷, 돈 아끼지 않는 감성 여행자 |
| W | A | L | N | 🌿 자연 아티스트 | 자연 속 인생샷, 좋은 곳엔 아낌없이 투자 |
| W | A | C | C | 📸 도심 사진작가 | 가성비 핫플에서 인생샷 건지는 알뜰 감성러 |
| W | A | C | N | 🥾 뚜벅이 탐험가 | 자연 속 알뜰 여행, 걸으며 사진 찍는 탐험가 |
| W | E | L | C | 🎯 도심 체험러 | 핫한 곳 직접 경험, 사진보다 기억이 중요 |
| W | E | L | N | 🌲 자연 힐링러 | 자연 속 고급 휴식, 눈으로 담는 여유 |
| W | E | C | C | 🚶 알뜰 도시 여행자 | 돈 아끼며 도심 구석구석 탐방 |
| W | E | C | N | 🌾 자연 탐방가 | 가성비 자연 여행, 걷기 좋아하는 실속파 |
| S | A | L | C | ✨ 럭셔리 인플루언서 | 고급 도심 숙소에서 인생샷, 여행=콘텐츠 |
| S | A | L | N | 🏕️ 프리미엄 감성 캠퍼 | 자연 속 고급 글램핑, SNS 감성 충전 |
| S | A | C | C | 📱 알뜰 콘텐츠 크리에이터 | 가성비 숙소에서도 인생샷 건지는 SNS러 |
| S | A | C | N | 🌅 자연 감성 사진가 | 자연 속 여유로운 인생샷, 알뜰하게 |
| S | E | L | C | 👑 도심 호캉스 귀족 | 고급 도심 호텔에서 완전 휴식, 사진은 패스 |
| S | E | L | N | 🧘 프리미엄 자연 힐러 | 자연 속 고급 리조트에서 진짜 힐링 |
| S | E | C | C | 🛋️ 가성비 도심 휴식러 | 저렴하게 도심 숙소에서 충전하는 현실파 |
| S | E | C | N | 🌙 자연 속 은둔자 | 저렴한 자연 속 숙소에서 조용히 쉬는 내향인 |

> 분류 기준: 각 축 점수 ≥ 50이면 고점 유형(W/A/L/C(City)), < 50이면 저점 유형(S/E/C(Cost)/N)

**결과 카드 예시:**

```
┌──────────────────────────────────┐
│      🥾 뚜벅이 탐험가            │
│                                  │
│     활동성 ████████░░ 85         │
│     사진   ███████░░░ 70         │
│     예산   ███░░░░░░░ 30         │
│     자연   ██████░░░░ 60         │
│                                  │
│  [인스타 공유] [카톡 공유]        │
└──────────────────────────────────┘
```

**비기능 요구사항:**
- 비회원 웹 참여 필수: 방장만 가입하고, 동행자는 공유 링크로 가입 없이 검사 가능
- 모바일 웹 최적화: 동행자 대부분이 카톡 인앱 브라우저로 접속

### 4.2 화면 2: 그룹 갈등 지도

**목적:** 동행자들의 TPTI를 겹쳐 보여주며, 어디서 충돌하고 어디서 일치하는지 시각화한다.

**기능 상세:**

| ID | 기능 | 설명 | 우선순위 |
|----|------|------|:---:|
| F2-1 | 여행 방 생성 | 방장이 방을 만들고 공유 링크 생성 | P0 |
| F2-2 | 동행자 참여 | 공유 링크로 접속 → TPTI 검사 완료 시 자동 합류 | P0 |
| F2-3 | 레이더 차트 겹치기 | 동행자들의 4축 레이더 차트를 하나의 그래프에 오버레이 | P0 |
| F2-4 | 충돌 축 하이라이트 | 편차가 큰 축을 빨간색으로 강조 표시 | P0 |
| F2-5 | 공통 지대 표시 | 모두가 유사한 점수를 보이는 축을 초록색으로 표시 | P0 |
| F2-6 | 충돌 요약 메시지 | "A님과 B님은 활동성에서 70점 차이로 충돌합니다" 자연어 요약 | P0 |
| F2-7 | 그룹 만족도 예측 | 현재 그룹 구성으로 예상되는 만족도 사전 안내 | P1 |

**충돌 판정 기준:**

| 편차 범위 | 충돌 등급 | 시각적 표시 |
|----------|----------|------------|
| 0~20 | 일치 (공통 지대) | 🟢 초록 |
| 21~40 | 경미한 차이 | 🟡 노랑 |
| 41~60 | 조율 필요 | 🟠 주황 |
| 61~100 | 심각한 충돌 | 🔴 빨강 |

### 4.3 화면 3: AI 합의 일정

**목적:** 갈등 지도를 바탕으로 모든 동행자가 납득할 수 있는 최적 일정을 AI가 생성한다.

**기능 상세:**

| ID | 기능 | 설명 | 우선순위 |
|----|------|------|:---:|
| F3-1 | 여행 조건 입력 | 여행 날짜, 일수(MVP: 1일), 지역 선택 | P0 |
| F3-2 | AI 일정 선택지 생성 | 합의 엔진이 특성이 다른 3가지 옵션 생성 (목표: 5초 이내) | P0 |
| F3-3 | 옵션별 개인 만족도 표시 | 각 옵션에 동행자별 예상 만족도(%) 표시 (예: "A님 72% / B님 68% / C님 61%") | P0 |
| F3-4 | 옵션 선택 및 확정 | 그룹이 하나의 옵션을 선택하여 일정 확정 | P0 |
| F3-5 | 슬롯별 취향 태그 | 각 슬롯이 누구의 취향을 반영했는지 표시 | P0 |
| F3-6 | 인구감소지역 배지 | 슬롯 장소가 인구감소지역이면 "숨은 명소" 배지 표시 | P0 |
| F3-7 | 지역 발굴형 옵션 고정 | 3가지 옵션 중 하나는 항상 인구감소지역 숨은 명소 중심 구성 | P0 |
| F3-8 | 기여 카드 | 일정 확정 후 "이 여행으로 XXX 지역 방문에 기여했습니다" 카드 생성 | P1 |
| F3-9 | 지도 뷰 | 장소들을 지도 위에 동선으로 표시 | P1 |

## 5. 합의 알고리즘 명세

### 5.0 일정 옵션 유형

TripSync는 단일 최적 일정이 아닌 특성이 다른 3가지 선택지를 생성한다.

| 옵션 | 유형 | 설명 |
|------|------|------|
| A | 균형형 | 그룹 최저 만족도를 최대화. 모두가 조금씩 만족 |
| B | 개성형 | 충돌 축을 시간 배분으로 교대 반영. 각자의 순간이 있음 |
| C | 지역 발굴형 | 인구감소지역 숨은 명소 중심 구성. 취향에 맞으면서 덜 알려진 장소 |

각 옵션에는 다음 정보가 표시된다:
- 동행자별 예상 만족도 (%)
- 일정 특성 태그 (균형형 / 개성형 / 지역 발굴형)
- 생성 근거 ("A님의 활동 취향을 오전에 배치, B님의 휴식 취향을 오후에 배치")

### 5.1 처리 흐름

```
[TPTI 벡터 입력]
       ↓
[Step A: 공통 지대 탐색]  ← 모든 동행자 편차 ≤ 20인 축 확정
       ↓
[Step B: 충돌 축 식별]    ← 편차 > 20인 축 추출
       ↓
[Step C: 시간 슬롯 배분]  ← 충돌 강도에 비례하여 교대 시간 배분
       ↓
[Step D: TourAPI 슬롯 매칭] ← 슬롯별 조건에 맞는 장소 필터링
       ↓
[Step E: 제약 조건 적용]  ← 거리(30분 이내), 운영시간 검증
       ↓
[Step F: 만족도 산출]     ← 기준 미달 시 Step C로 회귀
       ↓
[3가지 옵션 생성] → [그룹 선택] → [최종 일정 확정]
```

### 5.2 핵심 원리: "시간 배분 합의"

기존의 "평균값 추천" 방식을 사용하지 않는다. 예를 들어 A의 활동성이 90이고 B가 10이면 활동성 50짜리 장소를 추천하는 것이 아니라, **오전에는 A 취향(트레킹), 오후에는 B 취향(카페 휴식)처럼 시간을 나눠서 교대로 반영**한다.

핵심 원리: "모두가 조금씩 양보"가 아니라 **"모두가 돌아가며 만족"**이다.

### 5.3 시간 배분 규칙

| 규칙 | 내용 |
|------|------|
| 슬롯 크기 | 최소 1시간, 최대 3시간 |
| 이동 제약 | 이전 장소에서 다음 장소까지 30분 이내 |
| 운영시간 | 장소의 영업/운영 시간 내에만 배치 |
| 배분 비율 | 충돌 강도에 비례. 편차 70인 축은 편차 30인 축보다 더 많은 슬롯을 교대 배분 |
| 일정 범위 | MVP 기준 1일 (09:00~21:00) |

### 5.4 인구감소지역 연계 로직

TourAPI 장소 후보 선택 시 **인구감소지역 장소에 가중치를 부여**하여 자연스럽게 일정에 포함되도록 유도한다.

**앵커 + 숨은 명소 조합 원칙:**

```
[유명 앵커 장소 선택]  ← 동선의 구심점 (예: 공주 공산성)
       ↓
[반경 내 인구감소지역 장소 탐색]  ← 행안부 인구감소지역 시군 코드 기준
       ↓
[TPTI 취향 매칭 점수 계산]  ← 유사도 × 인구감소지역 가중치(+0.1)
       ↓
[슬롯에 배치]  ← 동선 연속성 유지 (직선거리 20km 이내)
```

**가중치 규칙:**

| 장소 유형 | 유사도 보정 | 비고 |
|---------|:---:|------|
| 일반 장소 | 기본값 | - |
| 인구감소지역 장소 | +0.10 | 행안부 89개 시·군 해당 |
| 인구감소지역 + 앵커 인근(20km) | +0.15 | 동선 연속성까지 만족 |

> 가중치는 취향 점수와 합산되므로, 취향과 동떨어진 장소가 억지로 추천되지 않는다. 자연스럽게 "취향에도 맞고, 인구감소지역이기도 한" 장소가 우선 선택된다.

### 5.5 만족도 점수 시스템

```
개인 만족도 = Σ(슬롯별 성향 일치도 × 슬롯 시간 비중)
그룹 만족도 = min(개인 만족도들)
```

- **최적화 기준:** 그룹 만족도(= 가장 불만족한 사람의 만족도)를 최대화
- **최소 임계값:** 그룹 만족도 ≥ 65%. 미달 시 자동 재조정
- **재조정 최대 횟수:** 3회. 3회 초과 시 현재 최선의 결과를 제시하고 사용자 수동 조정 유도

### 5.5 합의 엔진 LLM 프롬프트 구조

```
[System]
너는 여행 일정 합의 엔진이다. 아래 규칙을 따라라:
1. 공통 지대 축은 일정 전체의 기본 배경으로 설정한다.
2. 충돌 축은 시간 슬롯을 나눠 교대로 반영한다.
3. 각 슬롯은 최소 1시간, 최대 3시간이다.
4. 이전 장소에서 다음 장소까지 이동 시간 30분 이내여야 한다.
5. 결과에 각 동행자의 예상 만족도를 0~100%로 계산해서 표시한다.

[동행자 데이터]
(TPTI 벡터 삽입)

[공통 지대]
(분석 결과 삽입)

[충돌 축]
(분석 결과 + 시간 배분 지시 삽입)

[후보 장소 목록]
(TourAPI 캐시 DB에서 필터링된 장소 삽입)

위 조건으로 1일 일정(09:00~21:00)을 생성하라.
```

> **Hallucination 방지:** LLM은 일정의 구조(시간 배분, 순서)만 생성하고, 장소 데이터는 반드시 TourAPI 캐시 DB에서 가져온 후보 목록 내에서만 선택한다. 존재하지 않는 장소를 추천하는 것은 구조적으로 불가능하다.

## 8. 기술 아키텍처

### 8.1 기술 스택

| 레이어 | 기술 | 선정 이유 |
|--------|------|----------|
| 프론트엔드 | Next.js + Tailwind CSS | SSR로 카톡 링크 미리보기(OG) 지원, 빠른 프로토타이핑 |
| 백엔드 | NestJS (TypeScript) | 프론트와 언어 통일(TS), 모듈 기반 구조로 합의 엔진/TourAPI 모듈 분리 용이 |
| DB | MySQL | 관계형 데이터(사용자-그룹-일정) 구조에 적합, 팀 내 숙련도 확보 |
| AI | OpenAI GPT-4o / Google Gemini | 합의 엔진 프롬프트 처리. 두 모델 비교 테스트 후 품질/비용 기준으로 선택 |
| 데이터 | TourAPI + 자체 메타 태그 DB | 관광공사 데이터 활용 (공모전 필수 조건) |
| 프론트 배포 | Vercel | Next.js 최적화 배포, 무료 티어로 데모 운영 가능 |
| 백엔드+DB 배포 | AWS EC2 단일 인스턴스 | NestJS + MySQL을 하나의 EC2에 구성하여 인프라 비용 최소화 |

### 8.3 NestJS 모듈 구조

```

├── schedule/       # 일정 저장, 조회, 버전 관리
├── place/          # 장소 데이터 조회, 필터링

```

| Method | Endpoint | 설명 |

## 12. 로드맵

### Phase 1: MVP (공모전 출품) — 현재

- TPTI 8문항 검사 + 결과 카드
- 그룹 갈등 지도 시각화
- AI 합의 일정 생성 (1일, 충남권 한정)
- 인구감소지역 장소 가중치 + "숨은 명소" 배지
- 만족도 점수 시스템
- 카톡 공유 링크 (비회원 참여)

### Phase 2: 서비스 고도화

- 다중 지역 지원 (강원, 제주, 부산, 전남, 경북 등 수도권 제외 지방 중심)
- 전국 인구감소지역 데이터 확장 (89개 시·군 전체)
- 2박 3일 이상 일정 생성
- 실시간 상황 변경 ("체력 방전" → 근처 휴식 장소로 일정 변경)
- 네이버 플레이스 리뷰, 구글맵 평점 데이터 결합
- 예약 연계 (숙소/액티비티 Affiliate)

### Phase 3: 플랫폼 확장

- 여행 후 만족도 피드백 → 합의 알고리즘 학습 고도화
- 성향별 도시 추천 ("여행지 미정" 사용자 대응)
- 지자체 B2B 트렌드 리포트
- 프리미엄 구독 모델

## 13. 제약 사항 및 리스크

| 리스크 | 영향도 | 대응 방안 |
|--------|:---:|----------|
| TourAPI 일일 호출 제한 | 높음 | 일 1회 배치 수집 + 자체 DB 캐싱으로 실시간 호출 최소화 |
| LLM 응답 속도 지연 | 중간 | 로딩 UI 표시, 후보 장소를 사전 필터링하여 프롬프트 크기 최소화 |
| LLM 비용 증가 | 중간 | GPT-4o / Gemini 비교 테스트 후 비용 효율적인 모델 선택. 프롬프트 최적화 |
| 동행자 이탈률 | 높음 | 비회원 참여 + 8문항 30초 + 게이미피케이션 결과 카드 |
| TPTI 신뢰도 질문 (심사) | 중간 | "심리 검사가 아닌 대화 구조화 도구"로 포지셔닝. 사용자 수동 조정 기능 제공 |
| EC2 단일 장애점 | 낮음 | MVP 단계에서는 허용. Phase 2에서 RDS 분리 + Auto Scaling 적용 |
