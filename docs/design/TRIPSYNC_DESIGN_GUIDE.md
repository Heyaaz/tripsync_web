# TripSync Design Guide

이 문서는 현재 `app/page.tsx`와 `app/globals.css`에 구현된 TripSync 홈 디자인을 기준으로 정리한 실무용 가이드다.  
목적은 이후 화면을 추가하거나 기존 UI를 손볼 때, 같은 제품처럼 보이도록 시각 언어와 구현 규칙을 고정하는 데 있다.

## 1. Design Intent

TripSync의 홈 디자인은 다음 4가지 인상을 유지해야 한다.

- 밝고 가벼운 제품처럼 보일 것
- 여행 서비스답게 부드럽고 개방적인 분위기를 줄 것
- AI 기능을 과장하기보다 정돈된 신뢰감을 줄 것
- 한국어 중심 제품답게 큰 한글 타이포가 안정적으로 읽힐 것

한 줄로 요약하면 다음이다.

> 밝은 베이스 위에 유리 질감, 큰 한글 헤드라인, 절제된 블루 포인트, 둥근 카드와 부드러운 모션으로 완성하는 프리미엄 라이트 테마

## 2. Brand Personality

- 톤: 친절하지만 가벼워 보이지 않음
- 무드: clean, airy, soft-tech, premium
- 키워드: 합의, 조율, 여행 취향, 갈등 완화, 부드러운 신뢰
- 피해야 할 인상: 공격적인 AI SaaS, 네온, 다크모드 위주, 과한 미래감

## 3. Visual Principles

### 3.1 Bright Base First

- 전체 배경은 밝은 회백색 계열을 기본으로 사용한다.
- 가장 큰 면은 화이트 또는 매우 옅은 그레이로 유지한다.
- 강한 색은 포인트로만 쓴다.

### 3.2 One Primary Accent, Soft Supporting Accents

- 메인 포인트는 블루다.
- 보조 포인트는 바이올렛, 스카이, 오렌지, 에메랄드를 제한적으로 사용한다.
- 한 화면에서 모든 포인트 컬러를 동시에 크게 경쟁시키지 않는다.

### 3.3 Large Korean Headlines

- 홈의 핵심 인상은 큰 한글 헤드라인이다.
- 제목은 크기보다도 `font-black`, 좁은 자간, 타이트한 줄간격으로 존재감을 만든다.
- 한국어 텍스트에는 가능한 한 `break-keep-all`을 유지한다.

### 3.4 Rounded, Soft, Physical UI

- 카드, 버튼, 네비게이션 모두 라운드가 크다.
- 모서리가 날카로운 컴포넌트는 지양한다.
- 그림자는 짙고 무거운 드롭 섀도보다 넓고 부드러운 확산형 섀도를 쓴다.

### 3.5 Calm Motion

- 모든 인터랙션은 빠른 반응보다 “부드럽게 밀리는” 감각을 우선한다.
- `cubic-bezier(0.16, 1, 0.3, 1)`를 공통 시그니처로 유지한다.
- 이동량은 작게, opacity/transform 중심으로 제한한다.

## 4. Core Tokens

현재 구현 기준의 토큰은 아래 값을 우선 사용한다.

### 4.1 Color

#### Base

- `--bg`: `#F2F4F6`
- `--bg-2`: `#FFFFFF`
- `--bg-3`: `#F2F4F6`
- `--text-1`: `#191F28`
- `--text-2`: `#4E5968`
- `--text-3`: `#8B95A1`
- `--border`: `rgba(0, 0, 0, 0.04)`
- `--border-2`: `rgba(0, 0, 0, 0.08)`

#### Accent

- Primary Blue: `#3B82F6`
- Deep Blue Use Case: `#2563EB`
- Violet Support: `#7C3AED`
- Sky Support: `#0EA5E9`
- Emerald Support: `#10B981`
- Orange Support: `#EA580C` or `#F97316`
- Danger: `#EF4444`

#### Usage Rule

- 블루는 브랜드 주 색상이다.
- 바이올렛과 스카이는 gradient text나 보조 하이라이트에서만 사용한다.
- 오렌지와 에메랄드는 상태 구분 또는 서브 메시지 강조에 사용한다.
- 검정은 순수 `#000000` 대신 Zinc 계열의 짙은 중성색을 쓴다.

### 4.2 Radius

- `--radius-sm`: `10px`
- `--radius-md`: `14px`
- `--radius-lg`: `18px`
- `--radius-xl`: `24px`
- `--radius-2xl`: `32px`
- `--radius-full`: `9999px`

#### Radius Rule

- 입력/버튼: `md` 또는 `xl`
- 일반 카드: `xl`
- 강조 카드/대형 섹션: `2xl`
- 플로팅 네비/필/배지: `full` 또는 `2xl`

### 4.3 Shadow

기본 방향은 “짙은 그림자”가 아니라 “넓고 얕은 그림자”다.

- Small Surface: `0 2px 12px rgba(0,0,0,0.06)`
- Mid Card: `0 4px 24px rgba(0,0,0,0.05)`
- Lifted Card: `0 24px 60px -12px rgba(0,0,0,0.10)`
- Primary CTA: `0 8px 28px rgba(0,0,0,0.20)`
- Glass Nav: `0 2px 24px rgba(0,0,0,0.08)`

### 4.4 Motion

- Primary easing: `cubic-bezier(0.16, 1, 0.3, 1)`
- Hover lift: `translateY(-2px)` 또는 `translateY(-5px)`
- Active press: `scale(0.97)` 또는 `scale(0.98)`
- Reveal distance: 약 `20px` 내외
- Reveal delay step: `80ms`

## 5. Typography

### 5.1 Font

- 기본 폰트는 `Pretendard`
- 시스템 폴백은 `-apple-system`, `BlinkMacSystemFont`, `system-ui`, `sans-serif`

### 5.2 Weight Strategy

- `font-black`: hero headline, section headline, 핵심 숫자
- `font-bold`: 버튼, 카드 제목, 네비 주요 액션
- `font-medium`: 보조 설명, 부가 카피

### 5.3 Type Scale

현재 홈 기준 권장 스케일:

- Hero H1: `52 / 68 / 80`
- Section H2: `30 / 36 / 42` 전후
- Card Title: `18`
- Body Large: `17~18`
- Body Default: `15`
- Small Label: `11~13`

### 5.4 Korean Copy Rules

- 한국어 본문은 `leading-relaxed` 또는 `1.6+` 계열 유지
- 제목은 `tracking-tight` 사용
- 긴 문단은 너무 넓게 두지 말고 `max-w-md` 또는 유사 폭으로 제한
- 문장은 부드럽게 설명하되, 광고 문구처럼 과장하지 않는다

### 5.5 Legibility Rules

최근 앱 화면 정리 기준에서 가장 중요하게 추가된 원칙은 "작은 회색 텍스트도 흐려 보이지 않아야 한다"는 점이다.

- 작은 보조문장은 기본적으로 `text-zinc-700`을 우선 사용한다.
- `text-zinc-500`, `text-zinc-600`은 꼭 약하게 보여야 하는 메타 정보에만 제한적으로 쓴다.
- 작은 텍스트가 흐려 보인다면 weight를 올리기보다 먼저 크기를 키우고 색을 진하게 한다.
- 작은 회색 보조문장의 기본값은 아래 조합을 우선 사용한다.

```tsx
className="text-sm font-normal text-zinc-700 leading-relaxed"
```

- `11px~12px` 수준의 작은 텍스트는 특별한 이유가 없으면 피한다.
- uppercase + 넓은 tracking은 작은 한글 텍스트를 더 퍼져 보이게 만들 수 있으므로 최소화한다.
- 보조문장이 뭉개져 보일 때는 아래 순서로 수정한다.

1. 크기를 `text-sm`로 올린다.
2. 색을 `zinc-700` 이상으로 올린다.
3. `font-medium/semibold`보다 `font-normal`이 더 또렷한지 확인한다.
4. 그래도 흐리면 배경 surface의 투명도와 blur를 줄인다.

## 6. Layout System

### 6.1 Container

- 홈 메인 컨테이너는 `max-w-5xl`
- 일반 문서형/폼형 화면은 더 좁아도 되지만, 랜딩 계열은 과도하게 좁히지 않는다
- 기본 가로 패딩은 모바일 `px-6`, 소형 이상 `sm:px-8`

### 6.2 Vertical Rhythm

- Hero는 `min-h-[100dvh]`
- 주요 섹션 간 간격은 `mb-16`, `mb-32`처럼 크게 둔다
- 헤더와 본문은 붙이지 말고 충분히 띄운다

### 6.3 Section Pattern

현재 홈에서 재사용 가능한 주요 섹션 패턴:

- Floating Glass Nav
- Centered Hero
- Feature Pillar Strip
- 3-Step Card Grid
- Highlight Feature Card
- Full-Bleed CTA Block
- Minimal Footer

같은 페이지에서 모든 섹션을 같은 카드 패턴으로 반복하지 않는다.  
인접 섹션은 최소한 배치 방식 하나는 달라야 한다.

## 7. Component Guidelines

### 7.1 Floating Glass Navigation

특징:

- 화면 상단에서 살짝 떨어진 floating 구조
- 반투명 화이트 + 강한 blur
- 알약형 또는 라운드 큰 형태
- 브랜드, 보조 링크, 핵심 CTA만 남긴 최소 구성

규칙:

- 상단에 “붙은 바”처럼 만들지 않는다
- 네비 자체가 무겁거나 높아지지 않도록 한다
- 링크 수를 늘리기보다 중요한 행동 1개를 강조한다

### 7.2 Hero

특징:

- 큰 한글 제목
- 한 줄 강조를 위한 gradient text 가능
- 1차 CTA + 2차 CTA
- 바로 아래에 기능 요약 strip 배치 가능

규칙:

- gradient text는 페이지당 1회 수준으로 제한
- 핵심 문장은 짧고 강하게
- 보조 문장은 최대 2줄 정도가 이상적
- CTA는 “생성/시작/진단”처럼 행동이 분명한 단어를 쓴다

### 7.3 Feature Pillar Strip

특징:

- 반투명 또는 화이트 카드 한 덩어리 안에 3개 포인트 요약
- 아이콘 + 라벨 + 짧은 서브텍스트
- 모바일에서는 세로 stack, 데스크톱에서는 가로 분할

규칙:

- 통계 위장용 숫자 대신 실제 기능 요약을 쓴다
- 아이콘 배경은 아주 옅은 색면으로만 처리한다
- 본문보다 구조 인식이 먼저 되도록 텍스트는 짧게 유지한다

### 7.4 Step Cards

특징:

- 3단계 설명 카드
- 카드마다 색상 포인트는 다를 수 있으나, 전체 구조는 통일
- step number는 크고 옅게 처리

규칙:

- 카드 3개가 모두 같은 크기여도 시각적 포인트는 다르게 준다
- 배경 orb, border tint, CTA text color 정도만 변화시킨다
- 카드 내부 계층은 아이콘 → 번호 → 제목 → 설명 → 링크 순서를 유지한다

### 7.5 Highlight Feature Card

특징:

- 홈 중간의 큰 white card
- 왼쪽은 제품 가치 설명, 오른쪽은 기능 목록 패널
- 섹션 내부에 가벼운 radial tint 사용 가능

규칙:

- 이 카드가 페이지에서 제품 신뢰를 가장 강하게 설명해야 한다
- 우측 기능 패널은 3개 전후로 제한
- 카드 내부에서 또 복잡한 카드 중첩을 만들지 않는다

### 7.6 CTA Block

특징:

- 강한 블루 계열 배경
- 대비가 큰 화이트 CTA
- 간결한 헤드라인

규칙:

- 페이지 마지막 CTA는 주변 섹션보다 시각적 대비가 분명해야 한다
- CTA 버튼은 최소 2개까지, 그 이상은 산만하다
- 배경 그래디언트는 깊이를 주되 네온처럼 보이지 않게 한다

### 7.7 Footer

특징:

- 최소 정보만 제공
- 로고, 카피라이트, 필수 링크/소셜 수준에서 마감

규칙:

- 홈 푸터는 링크 농도를 높이지 않는다
- 제품의 여백감과 라이트한 분위기를 해치지 않도록 단정하게 유지한다

## 8. Surface & Background Rules

### 8.1 Background

- 기본 배경은 밝은 단색 + 아주 약한 ambient blob 조합
- 블롭은 장식이므로 본문보다 먼저 보이면 안 된다
- blur 강도는 높아도 opacity는 낮게 유지한다

### 8.2 Glass

- glass 효과는 네비게이션이나 소수의 강조 surface에 한정
- `bg-white/70~95`
- `backdrop-blur-xl` 이상 사용 가능
- border와 inset highlight를 함께 써서 질감을 만든다

추가 규칙:

- 앱형 화면에서는 glass를 과도하게 쓰지 않는다.
- 본문 카드, 폼 카드, 옵션 카드 같은 정보 중심 surface는 거의 불투명한 흰색을 우선한다.
- 텍스트 선명도가 떨어지면 blur를 줄이고 `bg-white` 또는 `bg-white/95+`로 올린다.
- 배경의 blob, radial tint, noise는 텍스트보다 먼저 인지되면 안 된다.

### 8.3 Noise

- 아주 약한 noise overlay는 허용
- 정보 전달보다 먼저 보이면 안 된다
- opacity는 0.03 수준처럼 매우 낮게 유지한다

## 9. Interaction Rules

### 9.1 Hover

- hover는 크기보다 깊이 변화 중심
- 카드: 약한 위로 lift
- 버튼: 약한 lift + shadow 강화
- 링크: 색 변화 또는 화살표 방향 유도

### 9.2 Active

- 모든 주요 버튼은 눌림 상태가 있어야 한다
- 과한 bounce 대신 작은 `scale`로 처리한다

### 9.3 Reveal

- 등장 애니메이션은 fade + translateY 조합
- 단계별 reveal delay는 `80ms` 간격 유지
- 모든 요소를 과하게 애니메이션하지 않는다

### 9.4 Proposal Vs Confirmed

TripSync 일정 화면은 "제안안"과 "확정안"을 시각적으로 명확히 구분해야 한다.

- 옵션 선택 단계는 "확정된 타임라인"처럼 보이면 안 된다.
- 제안안에서는 정확한 시간표보다 코스 흐름, 취향 반영 근거, 제안 메모를 중심으로 보여준다.
- `전체 타임라인 보기` 같은 확정형 표현보다 `제안 구성 보기`, `참고안`, `선택 후 조정` 같은 표현을 쓴다.
- 세로 타임라인, 촘촘한 시간 배지, 확정 일정처럼 보이는 강한 구조는 옵션 단계에서 지양한다.
- 확정 단계에서만 상세 시간과 실제 타임라인 구조를 강조한다.

## 10. Responsive Rules

- 모바일 우선으로 설계한다
- 모바일에서는 가로 나열보다 세로 stack을 우선한다
- 홈의 CTA 그룹, feature strip, 큰 카드 내부 2단 구성은 모바일에서 모두 세로로 무너져야 한다
- 고정 네비는 모바일에서도 한 줄에 유지하되, 요소 수를 늘리지 않는다
- 한국어 헤드라인은 작은 폭에서 잘리는 대신 자연스럽게 줄바꿈되도록 설계한다

추가 규칙:

- 작은 회색 보조문장은 모바일에서 더 뭉개져 보이므로 모바일 기준으로 먼저 읽힘을 확인한다.
- 모바일에서 보조문장, 주소, 메타 정보는 `text-sm` 아래로 쉽게 내리지 않는다.

## 11. Iconography

- 아이콘은 Solar 계열 유지
- 아이콘 크기는 작게, 배경 라운드 박스로 정리
- 아이콘 자체보다 텍스트 계층이 먼저 읽혀야 한다
- 장식용 아이콘은 본문 대비를 낮춰 사용한다

## 12. Copywriting Rules

- 문장은 문제 해결형으로 쓴다
- “AI가 알아서 해준다”보다 “갈등을 줄이고 합의 일정을 만든다”에 초점을 둔다
- 과장된 AI 마케팅 표현은 피한다
- CTA는 명령형이되 부담스럽지 않아야 한다

권장 예시:

- 여행방 만들기
- 내 여행 성향 알아보기
- 갈등 지점 미리 파악
- 모두가 만족할 일정 만들기

피해야 할 예시:

- 혁신적인 여행 AI 경험
- 차세대 스마트 플래닝
- 완벽한 초개인화 솔루션

## 13. Do / Don't

### Do

- 밝은 회백색 배경 위에 블루 포인트를 유지한다
- 큰 한글 타이포와 넓은 여백을 쓴다
- 카드, 버튼, 네비의 라운드 체계를 맞춘다
- 정보 카드는 짧고 빠르게 읽히게 만든다
- CTA 우선순위를 분명히 둔다

### Don't

- 다크모드를 기본 화면 톤으로 가져오지 않는다
- 네온 블루/퍼플 글로우를 과하게 쓰지 않는다
- 섹션마다 전혀 다른 디자인 시스템을 쓰지 않는다
- 얇고 날카로운 라운드, 작은 타이포, 촘촘한 레이아웃을 쓰지 않는다
- 제품과 무관한 장식 요소를 과도하게 추가하지 않는다

## 14. Implementation Guidance

새 페이지를 만들 때는 아래 순서를 우선 적용한다.

1. `globals.css` 토큰과 radius 체계를 그대로 쓴다.
2. 컨테이너 폭과 여백을 홈과 유사한 밀도로 맞춘다.
3. 헤드라인은 큰 한글 타입 스케일을 먼저 잡는다.
4. 버튼은 primary/secondary 위계를 명확히 둔다.
5. 카드에는 옅은 border와 넓은 그림자를 사용한다.
6. 한 화면에서 accent color의 중심은 블루로 유지한다.
7. 애니메이션은 reveal, hover, active 정도로 제한한다.

앱형 화면에서는 아래를 추가로 적용한다.

8. 작은 설명문은 `text-sm font-normal text-zinc-700`을 기본으로 시작한다.
9. 카드가 많아질수록 반투명도보다 가독성을 우선한다.
10. 제안 단계와 확정 단계는 같은 컴포넌트를 재사용하더라도 정보 밀도를 다르게 둔다.
11. 여행 날짜는 하루만 가정하지 말고 `시작일 ~ 종료일` 범위를 우선 고려한다.
12. AI가 제안하는 흐름과 사용자가 직접 추가하는 코스 편집 지점을 함께 설계한다.

## 15. Page Review Checklist

새 디자인을 적용하기 전 아래 항목을 확인한다.

- 배경이 충분히 밝고 여백이 넓은가
- 첫 화면에서 제품 가치가 한눈에 읽히는가
- CTA 1순위가 명확한가
- 컬러 중심이 블루 기반으로 정리되어 있는가
- 카드/버튼/네비 라운드가 같은 체계 안에 있는가
- 한국어 텍스트가 무겁지 않고 자연스럽게 읽히는가
- 섹션들이 모두 같은 구조 반복처럼 보이지 않는가
- 모션이 분위기를 살리되 산만하지 않은가
- 모바일에서 세로 흐름이 자연스러운가
- 작은 회색 보조문장이 흐리거나 번져 보이지 않는가
- blur나 반투명 surface 때문에 텍스트 대비가 죽지 않는가
- 일정 옵션 단계가 확정 일정처럼 보이지 않는가
- 날짜가 하루 기준으로 고정되지 않고 범위로 확장 가능하게 설계되어 있는가
- AI 제안 외에 사용자가 직접 조정하거나 추가할 지점이 있는가

## 16. Scope

이 가이드는 현재 홈 랜딩 디자인 기준 문서다.  
TPTI 결과 페이지, 방 생성 플로우, 일정 합의 화면 등 앱 내부 화면은 이 문서를 기반으로 하되, 정보 밀도와 사용성 요구에 맞게 더 단단한 앱 UI로 조정할 수 있다.

단, 아래 요소는 공통으로 유지한다.

- Pretendard 중심 타이포
- 밝은 중성 배경
- 블루 중심 포인트 컬러
- 큰 라운드와 부드러운 그림자
- 절제된 모션
- 작은 보조문장의 선명도 우선 원칙
- 제안안과 확정안의 시각적 구분
